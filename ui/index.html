Watch capacity move across a fleet

A live, interactive demo of BigFleet — a fleet-level infrastructure autoscaler. Press the buttons; the fleet on the right reacts in real time. We lead with the part a per-cluster autoscaler can’t do — then show how it decides, and prove it’s real.

1Not another cluster autoscaler

Cluster Autoscaler and Karpenter grow one cluster by bolting nodes onto it. BigFleet works a level up: it manages one finite pool of compute shared across many clusters and moves that capacity between them — reallocating one cluster’s freed-up committed capacity to serve unschedulable work in another, preempting low-priority work across the fleet under scarcity, and preferring cheap capacity over expensive. Same finite pool you already own, reallocated across clusters.

It provisions and reclaims nodes; it does not schedule pods — each cluster’s own kube-scheduler does that. This runs BigFleet’s real engine against a simulated cloud.

2Your fleet — three clusters, one finite pool

On the right: three real Kubernetes clusters — cluster-a, cluster-b, cluster-c — drawing on one shared, finite pool of machines. They’re already busy — running batch across the fleet plus a production workload on cluster-a — all packed onto already-paid-for committed capacity at $0/hr (no on-demand cloud running). The pool can’t grow past its ceiling, which is exactly why moving capacity between clusters is the whole game.

3Capacity moves between clusters

Here’s the payoff. You wind cluster-A’s production demand down and spike cluster-B’s — a routine traffic shift, and the committed pool is finite, so there’s no room to grow into. A per-cluster autoscaler would leave cluster-A’s freed nodes sitting idle and buy cluster-B a brand-new node. Watch BigFleet instead reclaim the nodes cluster-A no longer needs and hand that freed capacity to cluster-B — the same committed pool you already own, reallocated A→B: no new node, no new cloud spend on the move.

This cross-cluster move is BigFleet’s real control logic running against a simulated cloud. You changed the demand on both sides; BigFleet only reacts — as cluster-A’s demand falls it reclaims A’s freed nodes into the shard’s shared Idle pool, then reuses that committed hardware for cluster-B rather than bursting new cloud. It genuinely reasons about cost (the dollar figures are illustrative constants, not quotes). The wait you watch — cluster-A’s production draining, then cluster-B’s nodes coming up — is a simulated dwell, not a measured transfer time: BigFleet’s real decision, transfer speed simulated.

4A full fleet → preemption across the fleet

The other thing only a fleet-level autoscaler does. We fill the finite fleet, then a burst of critical production demand (PriorityClass critical) lands with no capacity left to provision. Priority decides: the real Kubernetes scheduler preempts running batch work to place the critical pods, and BigFleet reallocates the finite fleet toward what matters — the low-priority batch is what’s left waiting, not the work that counts.

Real Preempted events from the stock scheduler; real BigFleet reallocation. BigFleet doesn’t invent capacity — it allocates a finite fleet by priority. Only the provisioning time is simulated.

That’s the payoff — capacity moving across a finite fleet, the part no per-cluster autoscaler can reach. The rest is how it decides, and proof it’s all real.

5How it chooses: cheapest capacity first

Every move above is driven by cost: BigFleet scores candidate capacity by effective_cost = price + interruption_probability × penalty and fills the cheapest first. Three tiers, shared across all clusters:

• Committed — on-prem bare metal you own or reserved cloud. Already paid for, so work on it costs ~$0 at the margin. Filled first.

• On-demand cloud (GCP / AWS) — elastic, pay-per-use, stable; nothing until provisioned. Burst here when committed is full.

• Spot — the same SKUs, much cheaper, but interruptible. BigFleet routes interruption-tolerant work to cheap spot and interruption-sensitive work to stable on-demand — because spot’s interruption term outweighs the saving for that work. Real engine math, not a script.

Each tier is declared by a real CapacityProvider — three out-of-tree backends (on-prem / AWS / GCP) implementing the genuine providerkit six-RPC contract and passing the BigFleet conformance suite (core,cloud,spot); only the substrate (kwok) is simulated. The $ prices + interruption odds are provider-declared illustrative constants — the engine really reasons about them, but no measured saving is claimed.

6Drive it yourself: provision & reclaim

The table-stakes scale-out/in, shown so you can watch the cost ordering up close. Add a burst of production demand and BigFleet packs the pending requests onto fresh nodes — committed first (still $0), then bursting onto on-demand cloud ($ ticks up). Drop it and it sheds the billed cloud nodes first, keeping paid-for committed warm.

This is what any autoscaler does — it’s the fleet-level moves in §3–§4 that are BigFleet’s alone. Each node bin-packs several pods (open a dashboard to see); the provisioning time is a simulated dwell.

7The fleet is finite

No infinite cloud here. Committed is fixed and the on-demand quota has a ceiling. Push demand onto all three clusters at once: committed fills, on-demand fills behind it, and cost climbs to its peak until nearly every machine is in use. That hard ceiling is what makes the moves and preemption above necessary — and what a per-cluster view never sees.

8Prove the clusters are real

Each cluster is a genuine Kubernetes cluster — a real apiserver, the stock kube-scheduler, and real controllers. Open its official Kubernetes dashboard (opens in a new tab) and you’ll see the real nodes BigFleet provisioned — committed on-prem bm-dc1-… boxes alongside cloud gke-… / ip-10-… instances tagged bigfleet.demo/billing: on-demand or spot, with their zones and regions. The nodes are kwok-simulated (tagged bigfleet.demo/simulated, kwok:// providerID) — only the cloud is fake, never the control plane.

And the capacity itself flows through real providers: the shard provisions every node by calling three out-of-tree CapacityProvider backends (on-prem / AWS / GCP) over the genuine six-RPC providerkit contract — the same contract a real cloud provider implements, conformance-certified (core,cloud,spot). The spot-vs-on-demand tier you see on each node is BigFleet’s real effective-cost decision read back from what the provider declared, not a label the demo painted on.

9What’s real, what’s simulated

● Real
  • BigFleet’s engine and its Phase 1/2/3 decisions — it sorts candidate capacity by effective cost (price + interruption×penalty), so it really does fill the $0-marginal committed tier first, route interruption-tolerant work to cheap spot and sensitive work to stable on-demand, and reclaim under pressure
  • The three CapacityProviders (on-prem / AWS / GCP) — real providerkit six-RPC backends that pass the BigFleet conformance suite; the four capacity types + prices + interruption probabilities are provider-declared
  • The real upstream Kubernetes scheduler per cluster — native priority ordering, preemption & bin-packing (MostAllocated), nothing hand-rolled
  • A real Kubernetes apiserver, scheduler & controllers per cluster
● Simulated
  • The clouds themselves — on-prem / GCP / AWS are labels on kwok-simulated nodes, not real instances (the providers mint kwok fakes, not VMs)
  • The $ prices and interruption probabilities (committed $0 marginal, on-demand ~$0.38, spot ~$0.11 / interruption ~30%) — illustrative author-chosen constants, not cloud quotes; the engine really reasons about them, but no measured saving is claimed
  • Provisioning & drain time — the move is real; the ~15-40s a node takes to come up (and the drain on reclaim) is a configured dwell
  • Node internals — pod requests/limits, the kube/system reserve (Allocatable < Capacity) and the cni/kube-proxy/node-exporter DaemonSet baseline are illustrative; kwok runs pods at their request, so there are no real CPU/memory usage metrics

Industry context — third-party, not measured here and not a BigFleet result: independent analysis (Datadog, State of Cloud Costs) finds enterprise Kubernetes fleets average ~18% CPU utilisation, 2–5× over-provisioning, and an estimated $50k–$500k per cluster per year of wasted spend. This demo shows the engine reasoning about that kind of trapped capacity at a deliberately small scale — it measures and recovers nothing.
