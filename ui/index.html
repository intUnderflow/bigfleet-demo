Watch infrastructure move

A live, interactive demo of BigFleet — a fleet-level infrastructure autoscaler. Read down the page and press the buttons; the fleet on the right reacts in real time.

1What BigFleet is

Most autoscalers add or remove pods, or grow a single cluster. BigFleet works a level up: it manages one finite pool of compute capacity shared across many clusters and moves that capacity to wherever it’s needed — provisioning nodes where high-priority demand lands, reclaiming idle nodes elsewhere, and preferring cheap capacity over expensive.

It provisions and reclaims nodes; it does not schedule pods — each cluster’s own scheduler does that. This demo runs BigFleet’s real engine against a simulated cloud.

2Your fleet

On the right are three clusters — cluster-a, cluster-b, cluster-c. Each is a real Kubernetes apiserver, pre-loaded with baseline workloads already running on nodes BigFleet provisioned for them.

3Three kinds of capacity: committed, on-demand, spot

BigFleet sorts capacity by cost — specifically by effective_cost = price + interruption_probability × penalty. What matters isn’t where a machine lives but what it costs to put a given workload on it. The fleet draws on three tiers, shared across all three clusters:

• Committed — on-prem bare metal you own or reserved cloud (reserved instances / savings plans). Already paid for, so putting work on it costs ~$0 at the margin. BigFleet fills this first.

• On-demand cloud (GCP / AWS) — elastic, pay-per-use, stable. You pay only while a node runs; the rest is headroom that costs nothing until provisioned.

• Spot — the same cloud SKUs, much cheaper, but interruptible. For interruption-tolerant demand BigFleet provisions cheap spot (lowest effective cost); for interruption-sensitive demand it provisions stable on-demand instead — because spot’s interruption term outweighs the saving for that work. That capacity trade-off is the real engine math, not a script. (Which pod lands on which node is then the stock kube-scheduler’s bin-packing — a separate layer.)

Each tier is declared by a real CapacityProvider: three out-of-tree providers (on-prem bare-metal, AWS, GCP) that implement the genuine providerkit six-RPC contract and pass the BigFleet conformance suite (core,cloud,spot) — only the substrate underneath is simulated (kwok). Committed is ~28 on-prem bare-metal-8vcpu + ~20 AWS reserved m6i.2xlarge ($0-marginal); on-demand & spot share the burst SKUs m5.2xlarge / n2-standard-8, differing only in price + interruption. At rest you’re entirely on committed capacity at $0/hr. This is the whole finite fleet — it can’t grow past its ceiling.

4Move demand → committed fills, then cloud bursts

Add a burst of production demand to cluster-A. The new pods request capacity the cluster doesn’t have, so they go pending; BigFleet packs their requests onto fresh nodes — roughly one node per step of demand — topping out the already-paid-for committed pool first, then bursting onto on-demand cloud. Watch the committed bar fill, nodes take a few moments to provision, and on-demand cloud spend tick up from $0.

The decision — committed-before-on-demand — is BigFleet’s real Phase-1 cheapest-first logic; each node bin-packs several pods (open a dashboard to see). This demand scales OUT (new nodes), it doesn’t evict anything — priority & preemption come later (§8). The provisioning time is a simulated dwell.

5Drop demand → cloud is shed first

Now remove that demand. BigFleet reclaims the freed nodes — and it sheds the billed on-demand cloud nodes first (cloud spend falls back toward $0), keeping the already-paid-for committed capacity warm in the shared pool.

6Capacity flows between clusters

Capacity follows demand across the fleet. Spike a different cluster and watch the freed capacity flow there instead.

7Saturate the finite fleet

A real platform doesn’t have infinite capacity — and neither does this one. Committed capacity is fixed, and the on-demand cloud quota has a ceiling too. Push demand onto all three clusters at once: the committed pool fills, the on-demand cloud pool fills behind it, and cloud spend climbs to its peak — until nearly every machine is in use.

Watch the committed bar max out, the cloud bar fill, and the cost rise — this is the condition where a fleet-level autoscaler earns its keep.

8Critical demand into a full fleet → preemption

This is the moment a finite fleet matters. The fleet is full, but a burst of critical production demand (PriorityClass critical) lands on cluster-A. With no capacity left to provision, priority decides: the real Kubernetes scheduler places the critical pods first, and BigFleet reallocates capacity toward them — so the lower-priority batch work (PriorityClass batch) is what’s left waiting. The fleet serves what matters most.

Watch the fleet fill to its cap, then the real Kubernetes scheduler preempt running batch pods (PriorityClass batch) to place the critical demand — native preemption, real Preempted events. Both the scheduler’s priority decision and BigFleet’s capacity reallocation are real; only the provisioning time is simulated. BigFleet doesn’t invent capacity — it allocates a finite fleet by priority.

9Prove the clusters are real

Each cluster is a genuine Kubernetes cluster — a real apiserver, the stock kube-scheduler, and real controllers. Open its official Kubernetes dashboard (opens in a new tab) and you’ll see the real nodes BigFleet provisioned — committed on-prem bm-dc1-… boxes alongside cloud gke-… / ip-10-… instances tagged bigfleet.demo/billing: on-demand or spot, with their zones and regions. The nodes are kwok-simulated (tagged bigfleet.demo/simulated, kwok:// providerID) — only the cloud is fake, never the control plane.

And the capacity itself flows through real providers: the shard provisions every node by calling three out-of-tree CapacityProvider backends (on-prem / AWS / GCP) over the genuine six-RPC providerkit contract — the same contract a real cloud provider implements, conformance-certified (core,cloud,spot). The spot-vs-on-demand tier you see on each node is BigFleet’s real effective-cost decision read back from what the provider declared, not a label the demo painted on.

10What’s real, what’s simulated

● Real
  • BigFleet’s engine and its Phase 1/2/3 decisions — it sorts candidate capacity by effective cost (price + interruption×penalty), so it really does fill the $0-marginal committed tier first, route interruption-tolerant work to cheap spot and sensitive work to stable on-demand, and reclaim under pressure
  • The three CapacityProviders (on-prem / AWS / GCP) — real providerkit six-RPC backends that pass the BigFleet conformance suite; the four capacity types + prices + interruption probabilities are provider-declared
  • The real upstream Kubernetes scheduler per cluster — native priority ordering, preemption & bin-packing (MostAllocated), nothing hand-rolled
  • A real Kubernetes apiserver, scheduler & controllers per cluster
● Simulated
  • The clouds themselves — on-prem / GCP / AWS are labels on kwok-simulated nodes, not real instances (the providers mint kwok fakes, not VMs)
  • The $ prices and interruption probabilities (committed $0 marginal, on-demand ~$0.38, spot ~$0.11 / interruption ~30%) — illustrative author-chosen constants, not cloud quotes; the engine really reasons about them, but no measured saving is claimed
  • Provisioning & drain time — the move is real; the ~15-40s a node takes to come up (and the drain on reclaim) is a configured dwell
  • Node internals — pod requests/limits, the kube/system reserve (Allocatable < Capacity) and the cni/kube-proxy/node-exporter DaemonSet baseline are illustrative; kwok runs pods at their request, so there are no real CPU/memory usage metrics

Industry context — third-party, not measured here and not a BigFleet result: independent analysis (Datadog, State of Cloud Costs) finds enterprise Kubernetes fleets average ~18% CPU utilisation, 2–5× over-provisioning, and an estimated $50k–$500k per cluster per year of wasted spend. This demo shows the engine reasoning about that kind of trapped capacity at a deliberately small scale — it measures and recovers nothing.
